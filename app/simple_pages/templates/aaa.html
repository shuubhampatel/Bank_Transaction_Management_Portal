Lets read about AAA testing


Arrange, Act, Assert is the term for the three logical components that we should consider while creating our tests.

- We set up our environment, i.e. everything we need to run our test code, in the Arrange section. This includes setting up objects, dependencies, and all of the test case's data.
- We run the test in the Act section by invoking the functions we built up in the previous stage.
- Finally, in the Assert section, we specify the test's pass criteria, which will fail it if they are not met, i.e. if the test's actual findings differ from what we expected.

Let's understand more about Arrange, Act, Assert by test code example image1 

def test_calculation_division_get_result_method():
    """Testing the Calculator Division"""
    tuple_list = (1, 2) # <--- Arrange
    calculation = Division.create(tuple_list) # <--- Act
    assert calculation.get_result() == 0.5 # <--- Assert
  • Arrange:As we discuss above in the Arrange part have definition, docsting and Arrangement. so here we setting up our as Arrange we created tuple_list.

  • Act: This part is very important part here we gave instruction what this test do by invoking the functions we built up in the previous stage. so here in the above code from the given tuple list it will call method create() from class Division and do division.

  • Assert: Finally assertion will check the returned result from get_result() method with 0.5 to check our division method/ functionally in our app is working properly or not. If not then at the time of checking it will pass the message as this test is expecting result as a 0.5.


Why testing is important? :

A test is a procedure that exercises a behavior to determine if the behavior functions correctly. There are several different kinds of tests, like unit tests, integration tests, or end-to-end tests, but all functional tests do the same basic thing: they try something and report PASS or FAIL.

Testing provides an empirical feedback loop for development. That’s how testing keeps us safe. With tests, we know when things break. Without tests, coding can be dangerous. We don’t want to deploy big ol’ bugs! In big applications without tests, developers have to invest a lot of time in finding the bug, but with the help of unit tests,we can easily identify that the test is expecting this output and our code is giving a different result, so the developer can directly focus on that part directly.
