Lets read about Git

So GIT is a free and open-source version control system (VCS) that allows software developers to collaborate and keep a complete history of their work, as well as a distributed version control system (DVCS). you are wondering if Git is a DVCS (distributed), so what is the difference between a centralized version control system and a distributed one? Let’s first compare it to the general concept and then continue with the operations/terms/commands of Git, such as branches, merges, commits, tags, and repositories in Git.

  • Centralized version control is easier to learn than distributed. If you are a beginner, you’ll have to remember all the commands for all the operations in DVCS, and working on DVCS might be confusing initially
  • DVCS is faster than CVCS because you don’t need to communicate with the remote server for each and every command. You do everything locally, which gives you the benefit of working faster than CVCS.
  • DVCS has the biggest advantage in that it allows you to work offline and gives flexibility. You have the entire history of the code on your own hard drive, so all the changes you make will be on your own server or to your own repository, which doesn’t require an internet connection, but this is not the case with CVCS.
  • Merge conflicts with other developers' code are less in DVCS. Because every developer works on their own piece of code, Merge conflicts are more common in CVCS in comparison to DVCS.
  • If the main server goes down or crashes in DVCS, you can still get the backup or entire history of the code from your local repository or server where the full revision of the code is already saved. This is not the case in the case of CVCS. There is just a single remote server that has the entire code history.

So below are the Git operations/terms/commands:

  • Repositories:
    Git Repositories contain different files for different versions of a project. These files are imported into the local server as we talked about DVCS above, so a VCS is used to create these versions and store them in a specific place. We call it a repository. To initialize your repository in your project, type the command git init and start adding your files.
  • Branches:
    A branch is a repository version that differs from the main working project. Most contemporary version control systems include this feature. There can be multiple branches in a Git project. These branches offer a graphical representation of your changes. When you want to add a new feature or fix a bug, you generate a new branch to summarize your modifications
  • commit:
    So this Git command, as its name suggests, commits your work. That means saving your work in repositories, but as we talked about before, to commit anything, you have to add those files, and then you can commit it. Otherwise, if you do commit directly, it will tell you nothing to commit. So every time you make a change, you have to tell it to the commit command, which will work or execute. At the time of execution, you have to give one flag with a comment, and that flag could be -m or -a, so it represents the only modified file, and -a represents all files you can use according to your needs. The commit command looks like this: git commit -m "this is a comment" and yes, before this you have to add those changes, so for that you need to execute the add command, which is git add.
  • merge:
    git merge is a command that joins two development histories together.That is, if you are currently working on two different branches and want to combine all of your work from both branches into one, git merge is a useful procedure to use.So to perform a merge, you need to last commit, then you need to checkout to that branch and then you have to execute the merge command. That would be git merge [commit id]
  • Tags:
    are specific reference points in git history. Git tags are used to capture a specific point in history that is further used to point to a released version. You can create a tag with git tag <-a feature1.1> (also known as annotated tags)
