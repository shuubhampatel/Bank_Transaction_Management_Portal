Lets read about SOLID

SOLID is an acronym that stands for five object-oriented design principles that help you make the software more understandable, easier to maintain, and easier to extend.Below are the examples from Calculator app about sold principles to understand by examples.

  • Single-responsibility: As the name suggests, single responsibility means a call should have only one job. In our calculator app, in the operation and calculation names, you can see that we have separate classes for each operation or calculation, which shows the single responsibility principle in the calculator app.
    class Division:
    """This is the Division class"""

    @staticmethod
    def divide(value_1, value_2):
        """ This is the Divide method"""
        try:
            return value_1 / value_2
        except ZeroDivisionError:
            return "you can't divide by zero"


class Multiplication:
    """ This is the Multiplication class"""

    @staticmethod
    def multiply(value_1, value_2):
        """ This is the add method"""
        return value_1 * value_2


class Addition:
    """ This is the addition class"""

    @staticmethod
    def add(value_1, value_2):
        """ This is the add method"""
        return value_1 + value_2


class Subtraction:
    """ This is the subtraction class"""

    @staticmethod
    def subtract(value_1, value_2):
        """ This is the add method"""
        return value_1 - value_2
  • Open-closed: This principle tells us function should open for extension but close for modifications. so in the calculator class we are coding to a interface named calculations for specific class and also in the calculations interface we are importing another interface called operations and all the operation will perform there. so basically Coding to an interface is an integral part of SOLID so that's how open-closed principle used in calculator app we are extending our interfaces and performing operations and those operations methods are closed for any modifications they just take values and return the results.

    class Division(Calculation):
    """Division calculation object"""

    def get_result(self):
        """get the Division results"""
        result = 1.0
        for value in self.values:
            result = Div.divide(result, value)
        return result
  • Liskov Substitution: A sub-class must be able to take the place of its super-class. When extending a class, we should keep in mind that to pass subclass objects in place of parent class objects without disrupting client code. The goal of this principle is to ensure that a sub-class can correctly take the place of its super-class.
  • Interface Segregation: Make fine grained interfaces that are client specific Clients should not be forced to depend upon interfaces or methods that they do not use. in our calculator app we have class Multiplication and that class in not depended on any other class so basically this principle is kind of same as Single-responsibility but here insted of method here interface do one and only one thing.

That means the high-level module must not depend on the low-level module, but they should depend on abstractions. according to our calculator app operations module is the low-level module because it will perform operations after important task such as object creation, converting tuples to list because thease are the high level modules so according to dependency Inversion Principle operations module is depend upon calculations/ high level module.

Design Patterns:
Design patterns are common solutions to challenging issues in software development. They're similar to pre-made blueprints that we can use to fix a repeated design issue in the code.
It have many benefits that overcome problem in software design. it defines a common langauge that helps team to communicate more efficiently.
Patterns are classified in 3 groups by their intent.
  • Creational patterns: These patterns provide various object creation mechanisms, which increase flexibility and reuse of existing code.
  • Structural patterns: These patterns explain how to assemble objects and classes into larger structures while keeping these structures flexible and efficient.
  • Behavioral patterns: These patterns are concerned with algorithms and the assignment of responsibilities between objects.
